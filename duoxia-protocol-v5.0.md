## 6. 异常处理

### 6.1 GitHub同步冲突
**场景**: 多方同时编辑，push冲突  
**处理**: 
1. `git pull --rebase` 先拉取所有更新
2. 解决冲突（保留所有方有效内容）
3. 重新提交推送

### 6.2 某虾长时间离线
**场景**: 超过30分钟无响应  
**处理**:
1. 来福虾标记任务状态为 ⏸️ 暂停
2. 按照替补规则分配给其他在线虾执行
3. 原执行虾上线后可查看任务记录，补充未完成部分

### 6.3 网络中断
**场景**: GitHub无法访问  
**备用方案**: 
1. 飞书群聊中转任务和结果
2. Frank（PM）人工协调同步

## 8. 工作韧性原则（三虾共同遵守）

### 核心原则
1. **专业优先** - 任务优先分配给最专业的虾执行
2. **随时替补** - 任何虾离线时，其他虾主动接替对应任务
3. **主动补位** - 发现对方超过5分钟未响应时，先执行再通知
4. **不推诿** - 遇到模糊任务边界，主动承担
5. **汇报状态** - 执行前/后更新状态，让所有人知道进度
6. **互不抢功** - 最终成果明确标注各虾贡献

### 补位触发条件
| 触发条件 | 行动 |
|---------|------|
| 对应负责虾超过5分钟未响应 | 主动询问或直接接替 |
| 对应负责虾明确离线 | 立即接替任务 |
| Frank 催促任务 | 主动承接，不等对应虾 |
| 任务存在依赖卡点 | 主动协助解决卡点 |

### 状态汇报格式
```markdown
【状态更新】
from：[虾名]
状态：进行中/已完成/已接替
任务：[任务名称]
备注：[对应虾离线，我已接替/任务卡点需要协助]
时间：HH:MM
```
