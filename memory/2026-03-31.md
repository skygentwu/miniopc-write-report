### 话题5：SAP与Aspen的API开放程度现状
用户询问SAP和Aspen目前的API化程度。

**SAP（高度API化）：**
- 演进路径：BAPI/RFC → SOAP → OData/REST → RAP框架
- 当前状态：S/4HANA Cloud几乎100%功能可通过API访问
- 核心技术：OData（RESTful标准）、RAP（Restful ABAP Programming）、BTP Integration Suite
- 平台支持：跨平台，任何HTTP客户端均可调用
- AI友好度：★★★★★ 极其友好，官方主推API-First架构

**Aspen（有限API化）：**
- 技术基础：Windows COM/OLE自动化接口（非开放标准）
- 支持语言：Python（win32com）、MATLAB（actxserver）、Excel VBA
- 可执行操作：读取/修改变量、运行模拟、重新初始化
- 关键限制：仅Windows平台、官方文档极少、图形系统封闭
- AI友好度：★★★☆☆ 可行但需技巧，社区驱动

**核心差异：** SAP已完全转型为开放API架构；Aspen仍停留在桌面软件时代的COM接口，虽可编程但非现代化API设计。

### 话题6：RESTful WebService → CLI/AI原生架构改造
用户询问如何将现有RESTful接口系统改造为CLI或AI原生架构。

**改造路径一：RESTful → CLI**
- 阶段1：API端点映射为CLI命令（orders create/get/update/delete）
- 阶段2：选型CLI框架（Python-Click/Typer、Node-Commander/OCLIF、Go-Cobra）
- 阶段3：改造点：认证迁移至配置文件、输出支持表格/JSON、增加--interactive模式、支持管道

**改造路径二：RESTful → AI原生（推荐MCP协议）**
- 阶段1：能力拆解为Tools（create_order、check_inventory等）
- 阶段2：实现MCP（Model Context Protocol），定义tools.json
- 阶段3：三种模式：
  - A. API包装器：AI Agent → RESTful API（快速改造）
  - B. Function Calling直连：LLM → 数据库（高性能）
  - C. RAG+Agent：知识库 → Agent规划 → Tools执行（复杂决策）
- 阶段4：核心改造：接口语义化（自然语言替代URL）、上下文管理、多步骤编排、人在回路（HITL）

**推荐架构：分层混合**
```
顶层：AI Agent（自然语言交互）
中层：CLI工具（命令封装）
底层：RESTful API（保持不变）
```

**决策建议：**
- 服务技术人员 → 先CLI化
- 服务业务用户 → 直接AI原生
- 最稳妥：CLI作为基础层，AI作为交互层
