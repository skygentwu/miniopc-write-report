**版本**: v3.0  
**代号**: 双虾协作  
**生效日期**: 2026-03-22  
**核心架构**: Q福虾(本地) + 来福虾(云端) + ModelRoute多模型网关

## 一、架构总览

### 1.1 核心概念

miniOPC v3.0 是 **"1个PM + 2个AI智能体 + N个大模型"** 的最小化协作架构：

```
┌─────────────────────────────────────────────────────────┐
│                      Frank (PM)                          │
│                   人类项目经理                            │
└─────────────────────────────────────────────────────────┘
                            ↓ 派发任务
┌─────────────────────────────────────────────────────────┐
│                   来福虾 (云端虾)                         │
│              ☁️ 调度中枢 · 24h在线                        │
│     职责: 任务分解 → 分配调度 → 汇总融合 → 统一输出        │
│     模型: Kimi K2.5 + kimi-search                        │
│     位置: 北京云端服务器                                  │
└─────────────────────────────────────────────────────────┘
                            ↓ 分配子任务
        ┌──────────────────┼──────────────────┐
        ↓                  ↓                  ↓
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│   Q福虾在线   │   │  Q福虾离线   │   │   来福虾    │
│  🟢 多模型    │   │   ⚫ 替补    │   │   自检     │
│   并行执行    │   │   模式       │   │   执行     │
└──────────────┘   └──────────────┘   └──────────────┘
        ↓
┌─────────────────────────────────────────────────────────┐
│              Q福虾 ModelRoute 多模型网关                  │
│                   💻 本地虾 · 间歇在线                     │
│  可调用的模型:                                           │
│  - 🤖 DeepSeek   → 深度推理、代码生成                     │
│  - 🔍 百度千问   → 中文知识、百科问答                     │
│  - 💬 Kimi      → 长文档总结、信息提炼                    │
│  - 🎯 MinMax    → 多模态、语音合成                        │
│  - 🌍 Google Gemini → 海外情报、多语言                    │
│  位置: 济南 · MacBook Pro 16                             │
│  系统: QClaw (腾讯) + VPN                                │
└─────────────────────────────────────────────────────────┘
                            ↓ 返回结果
┌─────────────────────────────────────────────────────────┐
│                   来福虾 (云端虾)                         │
│              汇总融合 → 质量检查 → 统一输出                │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                      Frank (PM)                          │
│                   接收最终成果                            │
└─────────────────────────────────────────────────────────┘
```

### 1.2 与 miniOPC v2.0 的对比

| 维度 | miniOPC v2.0 | miniOPC v3.0 (双虾架构) |
|------|--------------|------------------------|
| **角色定义** | 4人战队(PM+提示词工程师+分析师+情报员+设计师) | 2个AI智能体(来福虾+Q福虾) |
| **执行实体** | 多个独立模型实例 | 2个OpenClaw实例，内部集成多模型 |
| **通信方式** | 群聊@ + 文件中转 | GitHub文件同步 + 飞书@唤醒 |
| **协作模式** | 实时同步 | **异步协作**(容忍离线) |
| **容错机制** | 人工介入替补 | **自动降级**(Q福虾离线→来福虾替补) |
| **模型调度** | 人工分配 | **ModelRoute自动路由** |
| **海外能力** | Gemini偶尔断线 | **Q福虾VPN+Gemini稳定访问** |

## 二、角色定义与职责

### 2.1 Frank (PM) - 人类项目经理

**唯一人类角色**，负责：
- 提出任务需求
- 确认任务分解方案
- 质量验收最终成果
- 紧急情况下@唤醒Q福虾

### 2.2 来福虾 (云端虾) - 调度中枢

**核心定位**: 永不掉线的项目总指挥

**职责矩阵**:

| 阶段 | 具体工作 | 使用工具/模型 |
|------|----------|--------------|
| **任务接收** | 理解Frank需求，澄清模糊点 | Kimi K2.5 |
| **任务分解** | 拆分子任务，定义交付物 | Kimi K2.5 + 思维链 |
| **模型调度** | 根据任务类型分配给最优模型 | 路由决策表 |
| **Q福虾在线** | 发送任务给Q福虾并行执行 | GitHub文件写入 |
| **Q福虾离线** | **自动降级**，自己替补执行 | kimi-search等 |
| **结果汇总** | 收集各模型输出，去重整合 | Kimi K2.5 |
| **质量检查** | 验证完整性、一致性、准确性 | 自检清单 |
| **统一输出** | 格式化输出，交付给Frank | Markdown/PPT/Excel |

**核心能力**:
- ✅ 24小时在线，永不掉线
- ✅ 任务分解与提示词工程
- ✅ 实时搜索 (kimi-search)
- ✅ 多模型结果融合
- ✅ 自动降级执行

**局限性**:
- ❌ 无法访问本地文件
- ❌ 无法使用VPN/Gemini
- ❌ 无法操作浏览器

### 2.3 Q福虾 (本地虾) - 多模型网关

**核心定位**: 强大的本地算力中心 + 海外情报站

**ModelRoute 可调用模型**:

| 模型 | 核心优势 | 最佳任务场景 | 执行者 |
|------|----------|-------------|--------|
| **DeepSeek** | 推理能力强，代码生成 | 复杂分析、算法设计、代码编写 | Q福虾 |
| **百度千问** | 中文语料丰富 | 国内政策、百科问答、中文创作 | Q福虾 |
| **Kimi** | 200万字长上下文 | 长篇文档总结、报告提炼 | Q福虾 |
| **MinMax** | 多模态能力强 | 语音合成、图片生成、视频脚本 | Q福虾 |
| **Google Gemini** | 全球数据访问，多语言 | 海外情报、国际案例、外文翻译 | Q福虾 |

**职责矩阵**:

| 阶段 | 具体工作 | 触发条件 |
|------|----------|----------|
| **任务接收** | 从GitHub读取来福虾派发的任务 | 收到文件更新通知 |
| **模型选择** | 根据任务类型选择最优模型 | ModelRoute自动路由 |
| **并行执行** | 同时调用多个模型处理不同子任务 | 复杂任务分解后 |
| **结果返回** | 写入GitHub文件，标记完成 | 所有子任务结束 |
| **状态声明** | 声明"已完成，等待接力" | 每次回复后 |

**核心能力**:
- ✅ 多模型并行执行
- ✅ VPN访问Gemini
- ✅ 本地文件操作
- ✅ 浏览器控制

**局限性**:
- ❌ 间歇在线(MacBook合盖即离线)
- ❌ 无法像来福虾那样24小时待命

    │       ↓ YES

    │       ↓ NO

    │       ↓ YES

## 四、异步协作流程

### 4.1 标准任务流程 (Q福虾在线)

```
T+0min    Frank: @来福虾 "帮我写一份制造业AI报告"
          ↓
T+1min    来福虾: 任务分解
          - 子任务A: 海外案例 (Gemini)
          - 子任务B: 国内政策 (百度千问)
          - 子任务C: 技术架构 (DeepSeek)
          - 子任务D: 汇总融合 (来福虾)
          ↓ 写入GitHub
T+2min    Q福虾: 读取任务，开始执行
          ↓ 并行调用多模型
T+5min    Q福虾: 完成子任务A/B/C，写入GitHub
          ↓
T+6min    来福虾: 读取结果，汇总融合
T+8min    来福虾: 输出完整报告给Frank
```

### 4.2 降级任务流程 (Q福虾离线)

```
T+0min    Frank: @来福虾 "帮我写一份制造业AI报告"
          ↓
T+1min    来福虾: 检查Q福虾状态 → ⚫ 离线
          ↓ 自动降级
T+1min    来福虾: 自己执行
          - 用kimi-search搜国内新闻
          - 用Kimi K2.5分析总结
          - (海外部分标记为"待Q福虾补充")
T+10min   来福虾: 输出初步报告给Frank
          ↓
[稍后]    Q福虾上线后: 补充海外部分
          ↓
来福虾: 更新完整版报告
```

## 五、容错与降级机制

### 5.1 Q福虾状态检测

来福虾每30分钟检查一次Q福虾状态：
- 🟢 在线: 5分钟内有GitHub更新
- 🟡 忙碌: 有"执行中"标记
- ⚫ 离线: 30分钟无响应
- ⚪ 未知: 首次检查或异常

### 5.2 自动降级策略

| 场景 | 降级动作 | 执行者 |
|------|----------|--------|
| Q福虾离线 | 海外任务标记"待补充"，其余自己来福虾执行 | 来福虾 |
| 模型调用失败 | 切换备用模型，记录失败原因 | Q福虾 |
| 任务超时 | 输出中间结果，标记未完成部分 | 来福虾 |
| GitHub不可访问 | 切换飞书群聊中转 | 双方 |

### 5.3 PM介入触发条件

以下情况需要Frank(PM)人工介入：
- 任务需求不明确，需要澄清
- Q福虾离线超过2小时，任务紧急
- 来福虾替补执行后质量不达标
- 双方对任务理解出现分歧

## 六、典型任务示例

### 示例1: 行业研究报告 (复杂任务)

**Frank**: "帮我写一份《2026制造业AI应用报告》"

**来福虾执行**:
1. 分解任务:
   - 海外案例 → Q福虾(Gemini)
   - 国内政策 → Q福虾(百度千问)
   - 技术架构 → Q福虾(DeepSeek)
   - 数据可视化 → Q福虾(MinMax)
   - 汇总融合 → 来福虾

2. 派发任务给Q福虾

3. 接收Q福虾返回的各部分结果

4. 汇总融合，生成完整报告

5. 输出给Frank

### 示例2: 紧急查询 (简单任务，Q福虾离线)

**Frank**: "马上给我国内最新AI补贴政策"

**来福虾执行** (Q福虾离线):
1. 检测到Q福虾⚫离线
2. 自动降级，自己执行
3. 用kimi-search搜索最新新闻
4. 用Kimi K2.5总结提炼
5. 输出结果给Frank
6. 标记"海外政策部分待Q福虾补充"

### 示例3: 代码开发 (专业任务)

**Frank**: "写一个Python脚本，爬取某网站数据并生成Excel"

**来福虾执行**:
1. 分配给Q福虾(DeepSeek)
2. Q福虾用DeepSeek生成代码
3. Q福虾本地测试(浏览器+文件操作)
4. 返回代码+测试结果
5. 来福虾汇总，输出给Frank

## 七、协议与规范

### 7.1 双虾交接协议 v2.0

详见: `shuangxia-protocol-v2.md`

核心要点:
- GitHub文件为主通信渠道

### 7.2 提示词模板

**任务分解Prompt模板**:
```
你是miniOPC v3.0的调度中枢。请分解以下任务：

任务: {用户输入}

请输出:
1. 子任务列表（含建议执行模型）
2. 各子任务交付物定义
3. 预计完成时间
4. 依赖关系

可用模型:
- Gemini: 海外情报
- DeepSeek: 代码/推理
- 百度千问: 中文知识
- Kimi: 长文档总结
- MinMax: 多模态
```

**模型调用Prompt模板** (Q福虾用):
```
你是{模型名称}，请执行以下任务：

任务: {子任务描述}

要求:
- 输出格式: {格式}
- 字数限制: {字数}
- 截止时间: {时间}

【输出格式要求】
1. 文件开头第一行必须是："【{模型名} 输出】"
2. 使用标准Markdown格式
3. 关键结论用 **加粗** 标注
4. 数据用表格呈现
5. 不要输出"以下是..."等套话
```
