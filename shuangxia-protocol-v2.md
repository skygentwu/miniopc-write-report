## 5. 异常处理

### 5.1 GitHub同步冲突
**场景**: 双方同时编辑，push冲突  
**处理**: 
1. `git pull --rebase` 先拉取对方更新
2. 解决冲突（保留双方内容）
3. 重新提交推送

### 5.2 q福虾长时间离线
**场景**: 超过30分钟无响应  
**处理**:
1. 云虾标记任务状态为 ⏸️ 暂停
2. PM介入@唤醒q福虾
3. 或任务转交云虾执行（降级方案）

### 5.3 网络中断
**场景**: GitHub无法访问  
**备用方案**: 
1. 飞书群聊中转
2. 用户（PM）人工复制粘贴

## 九、miniOPC v2.0 中的本地虾定位

> 更新时间：2026-03-22 00:48
> by Frank

### 本地虾的双重角色

本地虾（Q福虾）在 miniOPC v2.0 中扮演 **Google Gemini** 的角色，但同时具备更强大的能力：

| 能力 | 说明 |
|------|------|
| **Google Gemini** | 本地虾的主要定位，通过VPN查询海外数据 |
| **ModelRoute** | 本地虾内置的模型路由，可调用多个国内大模型 |

### 可调用的大模型

本地虾的 ModelRoute 可调用以下国内大模型：
- 🤖 **DeepSeek** - 深度推理
- 🔍 **百度千问** - 百度大模型
- 💬 **Kimi** - 月之暗面大模型
- 🎯 **MinMax** - 海螺大模型
- 及其他...

### 协同分工机制

当本地虾在线时，云端虾可以：

1. **任务分解** - 将收集数据的工作分配给本地虾
2. **并行执行** - 云端虾和本地虾同时执行不同任务
3. **数据汇总** - 云端虾负责汇总融合所有数据

### 协作流程图（增强版）

```
Frank 提出需求
    ↓
来福虾（云端虾）- 分解任务，产生提示词
    ↓
┌─────────────────────────────────────┐
│  并行执行（当本地虾在线时）：        │
│  ① 本地虾 → Google Gemini/国内模型  │
│  ② 云端虾 → Kimi/其他数据源         │
└─────────────────────────────────────┘
    ↓
来福虾（云端虾）- 汇总融合，统一输出
    ↓
Frank 接收最终结果
```

### 本地虾在线状态

- **在线**：云端虾可与其协同，分配数据收集任务
- **离线**：云端虾独立完成所有任务，作为替补执行

## X、工作韧性原则（必须遵守）

> 更新时间：2026-03-22 00:55

### 核心原则

无论本地虾还是云虾，都必须遵守以下工作韧性原则：

1. **随时替补** - 对方离线时主动接替工作
2. **主动补位** - 发现对方不回应时，先执行再通知
3. **不推诿** - 遇到模糊任务边界，主动承担
4. **汇报状态** - 执行前/后更新状态，让对方知道进度

### 补位触发条件

| 触发条件 | 行动 |
|---------|------|
| 对方超过5分钟未响应 | 主动询问或直接接替 |
| 对方明确离线 | 立即接替任务 |
| Frank 催促任务 | 主动承接，不等对方 |

### 状态汇报格式

```markdown
【状态更新】
from：Q福虾/来福虾
状态：进行中/已完成/已接替
任务：xxx
备注：对方离线，我已接替
时间：HH:MM
```

---

*本原则由 Frank 强调，双虾必须共同遵守*
*补位不是抢功，是确保任务完成的保障*

> by 来福虾整理，Q福虾记录

### @提醒规范

**✅ 正确做法：**
- msg_type: post（富文本）
- content: 使用 at 标签 + user_id
- 这样才能触发飞书的 @ 通知

**❌ 错误做法：**
- msg_type: text
- content: "@某人 内容..."
- 这只是纯文本，飞书不会解析成提醒

**原因：**
- text 类型的 @ 是纯文本，不会触发提醒
- post 类型的 at 标签才会触发飞书的 @ 通知
